## 1. Quick Start

> 🐳 **Prefer Docker?** The easiest way to run the server is the container image
> — see **[DOCKER.md](DOCKER.md)** (multi-arch, config via environment variables).

### Download or build

**Pre-built binary (Windows):**
```
securechat-server-windows-amd64.exe
```

**Build from source (requires Go 1.21+ and GCC for SQLite):**
```bash
# Linux / macOS
CGO_ENABLED=1 go build -o securechat-server .

# Windows (requires MinGW-w64 GCC in PATH)
set CGO_ENABLED=1
go build -o securechat-server.exe .
```

### First run

```bash
# Copy the example config and edit it
cp config.toml.example config.toml

# Generate a secure JWT secret (required in production)
openssl rand -hex 32

# Run the server
./securechat-server              # Linux / macOS
./securechat-server.exe          # Windows
```

On first start with an empty database, the server prints a **bootstrap invite code** in the console. Use it to register the first administrator account.

```
╔══════════════════════════════════════════════════════╗
║          BOOTSTRAP INVITE CODE (first user)          ║
║                                                      ║
║  8b82d0ae4758b7e833db59d46dff2bf0                    ║
║                                                      ║
║  Valid for 30 days. Share securely.                  ║
╚══════════════════════════════════════════════════════╝
```

The server listens on `0.0.0.0:8080` by default. Connect clients to `http://<server-ip>:8080`.

## 2. Configuration Reference

All settings live in `config.toml`. The server falls back to safe defaults if the file is absent.

```toml
[server]
host = "0.0.0.0"        # Bind address. Use "127.0.0.1" behind a reverse proxy.
port = 8080             # Listening port.
tls  = false            # Set true to enable native TLS (see §4).
cert = ""               # Absolute path to the PEM certificate file.
key  = ""               # Absolute path to the PEM private key file.
mode = "private"        # Registration and federation mode (see §3).

[database]
path = "./data.db"      # SQLite database file path (relative or absolute).
                        # The directory must be writable by the server process.

[limits]
max_message_size     = 65536   # Max WebSocket frame size in bytes (64 KB default).
max_rooms_per_user   = 50      # Max group rooms a user can be a member of.
offline_ttl_hours    = 72      # How long to keep offline messages (hours).
max_offline_messages = 500     # Max queued offline messages per user.

[jwt]
secret   = "CHANGE_ME"   # HMAC-SHA256 signing key. Generate: openssl rand -hex 32
ttl_days = 30            # Token lifetime in days. Clients re-authenticate after expiry.

[turn]
enabled = false           # Enable TURN relay for voice calls behind symmetric NAT.
url     = ""              # TURN server URL, e.g. "turn:turn.example.com:3478"
secret  = ""              # Shared secret for time-limited credentials (coturn style).

[federation]
# Required only when mode = "mesh_private" or "mesh_public"
name        = "My Node"           # Human-readable node name shown to peers.
public_url  = "https://example.com" # Externally reachable URL. No trailing slash.
secret      = "CHANGE_ME"         # Peers must send this in X-Federation-Secret header.
admin_token = "CHANGE_ME"         # Protects the /api/v1/admin/federation/* endpoints.
```

## 3. Server Modes

Set `mode` in `[server]` to control who can register and whether federation is active.

| Mode | Registration | User Search | Federation |
|------|-------------|-------------|------------|
| `private` | Invite code required | Local users only | Disabled |
| `public` | Open to anyone | Local users only | Disabled |
| `mesh_private` | Invite code required | Local + federated peers | Enabled |
| `mesh_public` | Open to anyone | Local + federated peers | Enabled |

**Choosing a mode:**

- **`private`** — recommended for closed communities (family, team, organization). Users can only join with a code generated by an existing member.
- **`public`** — suitable for open communities or testing. Anyone with the server URL can create an account.
- **`mesh_private`** — like `private`, but your server is part of a larger network. Users on other servers can message your users and vice versa. Registration still requires an invite.
- **`mesh_public`** — fully open node in a federated mesh. Maximum reach, minimum friction.

### Generating invite codes

Any registered user can generate an invite code from the app (Profile → Generate invite code). Codes are single-use and expire in 48 hours.

## 4. TLS / HTTPS

**Production servers must use TLS.** Without it, messages travel unencrypted over the network (even though the content is E2E encrypted at the application layer, metadata such as IP addresses and connection timing is exposed).

### Option A — Native TLS (recommended for simple setups)

Set the certificate and key paths in `config.toml`:

```toml
[server]
port = 443
tls  = true
cert = "/etc/securechat/fullchain.pem"
key  = "/etc/securechat/privkey.pem"
```

The server will serve HTTPS directly on port 443. Clients connect to `https://your.domain.com`.

#### Getting a free certificate with Let's Encrypt (certbot)

```bash
# Install certbot
sudo apt install certbot

# Obtain a certificate (standalone — stop the server first)
sudo certbot certonly --standalone -d your.domain.com

# Certificates are saved to:
# /etc/letsencrypt/live/your.domain.com/fullchain.pem
# /etc/letsencrypt/live/your.domain.com/privkey.pem
```

**Auto-renewal:** certbot installs a systemd timer automatically. The server process needs to be restarted after renewal to pick up the new certificate. Add a renewal hook:

```bash
# /etc/letsencrypt/renewal-hooks/deploy/restart-securechat.sh
#!/bin/bash
systemctl restart securechat
```
```bash
chmod +x /etc/letsencrypt/renewal-hooks/deploy/restart-securechat.sh
```

#### Self-signed certificate (for internal/LAN use)

```bash
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem \
  -days 365 -nodes -subj "/CN=192.168.1.100"
```

> **Note:** clients connecting to a self-signed certificate will need to trust it manually or disable certificate validation. Only use this on private LANs.

### Option B — Reverse proxy (nginx)

Keep the server on `localhost:8080` (no TLS) and let nginx handle HTTPS. This is the standard approach for servers running multiple services.

```nginx
server {
    listen 443 ssl;
    server_name your.domain.com;

    ssl_certificate     /etc/letsencrypt/live/your.domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/your.domain.com/privkey.pem;

    # WebSocket support (required)
    location / {
        proxy_pass         http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection "upgrade";
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_read_timeout 3600s;   # Keep WebSocket connections alive
    }
}

server {
    listen 80;
    server_name your.domain.com;
    return 301 https://$host$request_uri;
}
```

With this setup, set in `config.toml`:
```toml
[server]
host = "127.0.0.1"
port = 8080
tls  = false
```

Clients connect to `https://your.domain.com`.

### Certificate file locations (reference)

| Path | Contents |
|------|---------|
| `/etc/letsencrypt/live/<domain>/fullchain.pem` | Certificate chain (cert + intermediates) |
| `/etc/letsencrypt/live/<domain>/privkey.pem` | Private key |
| `/etc/securechat/cert.pem` | Suggested location for self-signed cert |
| `/etc/securechat/key.pem` | Suggested location for self-signed key |

The server process must have **read** access to both files. If running as a non-root user:
```bash
sudo chown root:securechat /etc/letsencrypt/live/<domain>/privkey.pem
sudo chmod 640 /etc/letsencrypt/live/<domain>/privkey.pem
```

## 5. Federation Mesh

Federation allows multiple SecureChat servers to form a mesh. Users on any node in the mesh can search for and message users on any other node. The server routes messages transparently — clients don't need to do anything special.

### How it works

```
Server A  ──── S2S API ────  Server B
 (mesh_private)               (mesh_private)

User @A sends DM to User @B:
  1. A's server receives the message (encrypted, can't read it).
  2. @B is not found locally — A queries its peers.
  3. Server B confirms it has @B.
  4. A relays the encrypted message to B via POST /s2s/message.
  5. B delivers to @B if online, or queues it for offline delivery.
```

Message content is **always E2E encrypted** before leaving the client. The server-to-server relay only moves ciphertext — neither server can read the message.

### Step-by-step setup

**On both servers**, set the mode and federation config:

```toml
# config.toml on Server A
[server]
mode = "mesh_private"

[federation]
name        = "Server A"
public_url  = "https://server-a.example.com"
secret      = "secret_A_32chars_minimum_aaaaaaa"   # peers use this to call Server A
admin_token = "admin_token_for_A_aaaaaaaaaaaaaaaa"
```

```toml
# config.toml on Server B
[server]
mode = "mesh_private"

[federation]
name        = "Server B"
public_url  = "https://server-b.example.com"
secret      = "secret_B_32chars_minimum_bbbbbbb"   # peers use this to call Server B
admin_token = "admin_token_for_B_bbbbbbbbbbbbbbbb"
```

> Generate secrets with: `openssl rand -hex 32`

**Restart both servers**, then register each as a peer of the other:

```bash
# Tell Server A about Server B
# (use Server B's federation.secret in the "secret" field)
curl -X POST https://server-a.example.com/api/v1/admin/federation/peers \
     -H "X-Admin-Token: admin_token_for_A_aaaaaaaaaaaaaaaa" \
     -H "Content-Type: application/json" \
     -d '{
       "url":    "https://server-b.example.com",
       "name":   "Server B",
       "secret": "secret_B_32chars_minimum_bbbbbbb"
     }'

# Tell Server B about Server A
curl -X POST https://server-b.example.com/api/v1/admin/federation/peers \
     -H "X-Admin-Token: admin_token_for_B_bbbbbbbbbbbbbbbb" \
     -H "Content-Type: application/json" \
     -d '{
       "url":    "https://server-a.example.com",
       "name":   "Server A",
       "secret": "secret_A_32chars_minimum_aaaaaaa"
     }'
```

**Verify the peer list:**

```bash
curl -H "X-Admin-Token: admin_token_for_A_aaaaaaaaaaaaaaaa" \
     https://server-a.example.com/api/v1/admin/federation/peers
```

**Remove a peer:**

```bash
curl -X DELETE \
     "https://server-a.example.com/api/v1/admin/federation/peers?url=https://server-b.example.com" \
     -H "X-Admin-Token: admin_token_for_A_aaaaaaaaaaaaaaaa"
```

### Client experience

- Federated users appear in search results with a server badge showing their home server's hostname.
- The Profile tab shows the list of federated servers the current server is connected to.
- Messaging works identically to local messaging — the routing is transparent.

### Priority and failover

The client app fetches the federation list on every connection and stores it locally with a reliability score. The score factors in recency of last successful contact and consecutive failure count. Future reconnection logic uses this score to prefer more reliable servers.

## 6. TURN Server (Voice NAT traversal)

WebRTC voice calls use STUN by default (Google's public STUN: `stun:stun.l.google.com:19302`). STUN is sufficient for most home and office networks. TURN is needed when:

- Users are behind a **symmetric NAT** (common in corporate environments and some mobile carriers).
- STUN negotiation fails (connection drops after ~10 seconds with no audio).

### Installing coturn (Linux)

```bash
sudo apt install coturn

# Generate a TURN secret
openssl rand -hex 32
```

**`/etc/turnserver.conf` (minimal):**
```
listening-port=3478
fingerprint
lt-cred-mech
use-auth-secret
static-auth-secret=YOUR_TURN_SECRET_HERE
realm=turn.your.domain.com
log-file=/var/log/turnserver.log
```

```bash
sudo systemctl enable coturn
sudo systemctl start coturn
```

**Open firewall ports:** TCP/UDP 3478 (TURN), UDP 49152-65535 (relay ports).

### Configuring SecureChat to use TURN

```toml
[turn]
enabled = true
url     = "turn:turn.your.domain.com:3478"
secret  = "YOUR_TURN_SECRET_HERE"
```

The server issues time-limited TURN credentials to authenticated clients. Credentials rotate every session — the static secret never reaches the client.

## 7. Production Deployment (Linux)

### Create a dedicated user

```bash
sudo useradd --system --no-create-home --shell /usr/sbin/nologin securechat
sudo mkdir -p /opt/securechat
sudo chown securechat:securechat /opt/securechat
```

### Copy files

```bash
sudo cp securechat-server /opt/securechat/
sudo cp config.toml       /opt/securechat/
sudo chmod 700 /opt/securechat/securechat-server
sudo chmod 600 /opt/securechat/config.toml
```

### systemd service

```ini
# /etc/systemd/system/securechat.service
[Unit]
Description=SecureChat Server
After=network.target

[Service]
Type=simple
User=securechat
WorkingDirectory=/opt/securechat
ExecStart=/opt/securechat/securechat-server -config /opt/securechat/config.toml
Restart=on-failure
RestartSec=5

# Harden the process
NoNewPrivileges=yes
PrivateTmp=yes
ProtectSystem=strict
ReadWritePaths=/opt/securechat

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable securechat
sudo systemctl start securechat
sudo journalctl -u securechat -f   # Follow logs
```

### Firewall (ufw)

```bash
sudo ufw allow 443/tcp    # HTTPS (if using native TLS)
sudo ufw allow 80/tcp     # HTTP redirect (nginx only)
# Do NOT expose port 8080 publicly when using a reverse proxy
```

### Database backups

```bash
# SQLite online backup (safe while server is running)
sqlite3 /opt/securechat/data.db ".backup /opt/securechat/data.db.bak"

# Cron — daily backup at 03:00
echo "0 3 * * * securechat sqlite3 /opt/securechat/data.db \".backup /opt/securechat/backups/data-\$(date +\%Y\%m\%d).db\"" \
     | sudo crontab -u securechat -
```

## 9. Security Checklist

- [ ] **Change `jwt.secret`** — never use the default. Generate with `openssl rand -hex 32`.
- [ ] **Enable TLS** — either native (`tls = true`) or via nginx reverse proxy.
- [ ] **Set `mode = "private"`** unless you intentionally want open registration.
- [ ] **Change `federation.secret` and `federation.admin_token`** before enabling mesh mode.
- [ ] **Restrict port 8080** in the firewall if using a reverse proxy (only localhost should reach it).
- [ ] **Run as a non-root user** (Linux: the `securechat` system user from §7).
- [ ] **Back up `data.db`** regularly — it contains all user registrations and offline messages.
- [ ] **Monitor disk space** — SQLite grows with offline messages; they are purged after `offline_ttl_hours`.

## 10. API Reference

All endpoints except `/api/v1/health`, `/api/v1/register`, `/api/v1/federation`, and `/s2s/*` require a valid JWT in `Authorization: Bearer <token>`.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/api/v1/health` | None | Health check |
| `POST` | `/api/v1/register` | None | Register a new user |
| `POST` | `/api/v1/invites` | JWT | Generate an invite code |
| `GET` | `/api/v1/users/{id}` | JWT | Get user profile and public key |
| `GET` | `/api/v1/users?q=` | JWT | Search users (fans out to peers in mesh mode) |
| `POST` | `/api/v1/rooms` | JWT | Create a group room |
| `GET` | `/api/v1/rooms/{id}` | JWT | Get room info |
| `GET` | `/api/v1/rooms?q=` | JWT | Search rooms |
| `GET` | `/api/v1/ws?token=` | JWT (query) | Upgrade to WebSocket |
| `GET` | `/api/v1/federation` | None | Federation info and peer list |
| `POST` | `/api/v1/admin/federation/peers` | `X-Admin-Token` | Add a federation peer |
| `GET` | `/api/v1/admin/federation/peers` | `X-Admin-Token` | List federation peers |
| `DELETE` | `/api/v1/admin/federation/peers?url=` | `X-Admin-Token` | Remove a federation peer |
| `GET` | `/s2s/users/{id}` | `X-Federation-Secret` | S2S user lookup |
| `GET` | `/s2s/users?q=` | `X-Federation-Secret` | S2S user search |
| `POST` | `/s2s/message` | `X-Federation-Secret` | S2S message relay |

## 11. Troubleshooting

**Server won't start — "open database: ..."**
The database path is not writable. Check directory permissions and that the path exists.

**Clients can't connect — "invalid token"**
The `jwt.secret` in `config.toml` was changed after users registered. All existing JWTs are now invalid. Users must re-register or the old secret must be restored.

**WebSocket disconnects immediately**
Likely a proxy timeout. Set `proxy_read_timeout 3600s` in nginx (see §4).

**Voice calls fail with no audio**
STUN failed — try enabling TURN (§6). Also verify UDP ports 49152-65535 are open in the firewall.

**Federation — messages not arriving**
1. Verify both servers can reach each other: `curl -H "X-Federation-Secret: <secret>" https://peer/s2s/users/<some_id>`
2. Check that the `secret` in the peer record matches the remote server's `federation.secret`.
3. Both servers must be in mesh mode.

**"ws read error: websocket: close 1006 (abnormal closure)"**
This is a normal client disconnect (app closed, network drop). It is not an error. The server logs it only for unexpected close codes; 1006 is already suppressed in the current build.

## 3. Modos del Servidor

Establece `mode` en `[server]` para controlar quién puede registrarse y si la federación está activa.

| Modo | Registro | Búsqueda de usuarios | Federación |
|------|----------|---------------------|------------|
| `private` | Requiere código de invitación | Solo usuarios locales | Desactivada |
| `public` | Abierto a cualquiera | Solo usuarios locales | Desactivada |
| `mesh_private` | Requiere código de invitación | Locales + peers federados | Activada |
| `mesh_public` | Abierto a cualquiera | Locales + peers federados | Activada |

**Cómo elegir:**

- **`private`** — recomendado para comunidades cerradas (familia, equipo, organización). Solo entran con código generado por un miembro existente.
- **`public`** — para comunidades abiertas o pruebas. Cualquiera con la URL del servidor puede crear cuenta.
- **`mesh_private`** — como `private`, pero el servidor forma parte de una red mayor. Los usuarios de otros servidores pueden escribirte y viceversa. El registro sigue requiriendo invitación.
- **`mesh_public`** — nodo completamente abierto en una malla federada. Máximo alcance, mínima fricción.

### Generar códigos de invitación

Cualquier usuario registrado puede generar un código desde la app (Perfil → Generar código de invitación). Los códigos son de un solo uso y caducan a las 48 horas.

## 4. TLS / HTTPS

**Los servidores en producción deben usar TLS.** Sin él, aunque el contenido de los mensajes está cifrado extremo a extremo, los metadatos (IPs, tiempos de conexión) quedan expuestos.

### Opción A — TLS nativo (recomendado para configuraciones simples)

```toml
[server]
port = 443
tls  = true
cert = "/etc/securechat/fullchain.pem"
key  = "/etc/securechat/privkey.pem"
```

El servidor sirve HTTPS directamente en el puerto 443. Los clientes se conectan a `https://tu.dominio.com`.

#### Certificado gratuito con Let's Encrypt (certbot)

```bash
# Instalar certbot
sudo apt install certbot

# Obtener un certificado (modo standalone — detén el servidor primero)
sudo certbot certonly --standalone -d tu.dominio.com

# Los certificados quedan en:
# /etc/letsencrypt/live/tu.dominio.com/fullchain.pem
# /etc/letsencrypt/live/tu.dominio.com/privkey.pem
```

**Renovación automática:** certbot instala un timer de systemd. Añade un hook para reiniciar el servidor tras la renovación:

```bash
# /etc/letsencrypt/renewal-hooks/deploy/restart-securechat.sh
#!/bin/bash
systemctl restart securechat
```
```bash
chmod +x /etc/letsencrypt/renewal-hooks/deploy/restart-securechat.sh
```

#### Certificado autofirmado (uso en LAN/interno)

```bash
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem \
  -days 365 -nodes -subj "/CN=192.168.1.100"
```

> **Nota:** los clientes que se conecten a un certificado autofirmado necesitarán aceptarlo manualmente. Úsalo solo en redes privadas.

### Opción B — Proxy inverso (nginx)

Mantén el servidor en `localhost:8080` (sin TLS) y deja que nginx gestione HTTPS.

```nginx
server {
    listen 443 ssl;
    server_name tu.dominio.com;

    ssl_certificate     /etc/letsencrypt/live/tu.dominio.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/tu.dominio.com/privkey.pem;

    # Soporte WebSocket (obligatorio)
    location / {
        proxy_pass         http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection "upgrade";
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_read_timeout 3600s;   # Mantener conexiones WebSocket abiertas
    }
}

server {
    listen 80;
    server_name tu.dominio.com;
    return 301 https://$host$request_uri;
}
```

Con esta configuración, en `config.toml`:
```toml
[server]
host = "127.0.0.1"
port = 8080
tls  = false
```

Los clientes se conectan a `https://tu.dominio.com`.

### Ubicación de los certificados (referencia)

| Ruta | Contenido |
|------|---------|
| `/etc/letsencrypt/live/<dominio>/fullchain.pem` | Cadena de certificados (cert + intermedios) |
| `/etc/letsencrypt/live/<dominio>/privkey.pem` | Clave privada |
| `/etc/securechat/cert.pem` | Ubicación sugerida para certificado autofirmado |
| `/etc/securechat/key.pem` | Ubicación sugerida para clave autofirmada |

El proceso del servidor necesita permiso de **lectura** sobre ambos archivos. Si corre como usuario no-root:
```bash
sudo chown root:securechat /etc/letsencrypt/live/<dominio>/privkey.pem
sudo chmod 640 /etc/letsencrypt/live/<dominio>/privkey.pem
```

## 5. Malla Federada

La federación permite que varios servidores SecureChat formen una malla. Los usuarios de cualquier nodo pueden buscar y escribir a usuarios de cualquier otro nodo. El servidor enruta los mensajes de forma transparente.

### Cómo funciona

```
Servidor A  ──── API S2S ────  Servidor B
(mesh_private)                  (mesh_private)

Usuario @A envía un DM a Usuario @B:
  1. El servidor de A recibe el mensaje (cifrado, no puede leerlo).
  2. @B no existe localmente — A consulta a sus peers.
  3. El servidor B confirma que tiene a @B.
  4. A reenvía el mensaje cifrado a B mediante POST /s2s/message.
  5. B lo entrega a @B si está conectado, o lo encola offline.
```

El contenido siempre va cifrado extremo a extremo. El relay S2S mueve solo texto cifrado.

### Configuración paso a paso

**En ambos servidores**, configura el modo y la sección federation:

```toml
# config.toml del Servidor A
[server]
mode = "mesh_private"

[federation]
name        = "Servidor A"
public_url  = "https://servidor-a.ejemplo.com"
secret      = "secreto_A_32chars_minimo_aaaaaaaa"   # los peers usan esto para llamar a A
admin_token = "admin_token_para_A_aaaaaaaaaaaaaa"
```

```toml
# config.toml del Servidor B
[server]
mode = "mesh_private"

[federation]
name        = "Servidor B"
public_url  = "https://servidor-b.ejemplo.com"
secret      = "secreto_B_32chars_minimo_bbbbbbbb"   # los peers usan esto para llamar a B
admin_token = "admin_token_para_B_bbbbbbbbbbbbbb"
```

> Genera secretos con: `openssl rand -hex 32`

**Reinicia ambos servidores**, luego regístralos mutuamente como peers:

```bash
# Informar al Servidor A sobre el Servidor B
# (el campo "secret" es el federation.secret del SERVIDOR B)
curl -X POST https://servidor-a.ejemplo.com/api/v1/admin/federation/peers \
     -H "X-Admin-Token: admin_token_para_A_aaaaaaaaaaaaaa" \
     -H "Content-Type: application/json" \
     -d '{
       "url":    "https://servidor-b.ejemplo.com",
       "name":   "Servidor B",
       "secret": "secreto_B_32chars_minimo_bbbbbbbb"
     }'

# Informar al Servidor B sobre el Servidor A
curl -X POST https://servidor-b.ejemplo.com/api/v1/admin/federation/peers \
     -H "X-Admin-Token: admin_token_para_B_bbbbbbbbbbbbbb" \
     -H "Content-Type: application/json" \
     -d '{
       "url":    "https://servidor-a.ejemplo.com",
       "name":   "Servidor A",
       "secret": "secreto_A_32chars_minimo_aaaaaaaa"
     }'
```

**Verificar la lista de peers:**

```bash
curl -H "X-Admin-Token: admin_token_para_A_aaaaaaaaaaaaaa" \
     https://servidor-a.ejemplo.com/api/v1/admin/federation/peers
```

**Eliminar un peer:**

```bash
curl -X DELETE \
     "https://servidor-a.ejemplo.com/api/v1/admin/federation/peers?url=https://servidor-b.ejemplo.com" \
     -H "X-Admin-Token: admin_token_para_A_aaaaaaaaaaaaaa"
```

### Experiencia del usuario

- Los usuarios federados aparecen en los resultados de búsqueda con una etiqueta que indica el servidor de origen.
- La pestaña Perfil muestra la lista de servidores federados a los que está conectado el servidor actual.
- Los mensajes funcionan igual que los locales — el enrutamiento es transparente.

## 6. Servidor TURN (Voz con NAT)

Las llamadas de voz por WebRTC usan STUN por defecto. TURN es necesario cuando los usuarios están detrás de un **NAT simétrico** (común en entornos corporativos y algunas operadoras móviles).

### Instalar coturn (Linux)

```bash
sudo apt install coturn

# Generar un secreto TURN
openssl rand -hex 32
```

**`/etc/turnserver.conf` (mínimo):**
```
listening-port=3478
fingerprint
lt-cred-mech
use-auth-secret
static-auth-secret=TU_SECRETO_TURN_AQUI
realm=turn.tu.dominio.com
log-file=/var/log/turnserver.log
```

```bash
sudo systemctl enable coturn
sudo systemctl start coturn
```

**Abre en el firewall:** TCP/UDP 3478 (TURN) y UDP 49152-65535 (puertos de relay).

### Configurar SecureChat para usar TURN

```toml
[turn]
enabled = true
url     = "turn:turn.tu.dominio.com:3478"
secret  = "TU_SECRETO_TURN_AQUI"
```

El servidor emite credenciales TURN temporales a los clientes autenticados. Las credenciales rotan en cada sesión — el secreto estático nunca llega al cliente.

## 7. Despliegue en Producción (Linux)

### Crear un usuario dedicado

```bash
sudo useradd --system --no-create-home --shell /usr/sbin/nologin securechat
sudo mkdir -p /opt/securechat
sudo chown securechat:securechat /opt/securechat
```

### Copiar los archivos

```bash
sudo cp securechat-server /opt/securechat/
sudo cp config.toml       /opt/securechat/
sudo chmod 700 /opt/securechat/securechat-server
sudo chmod 600 /opt/securechat/config.toml
```

### Servicio systemd

```ini
# /etc/systemd/system/securechat.service
[Unit]
Description=SecureChat Server
After=network.target

[Service]
Type=simple
User=securechat
WorkingDirectory=/opt/securechat
ExecStart=/opt/securechat/securechat-server -config /opt/securechat/config.toml
Restart=on-failure
RestartSec=5

# Endurecimiento del proceso
NoNewPrivileges=yes
PrivateTmp=yes
ProtectSystem=strict
ReadWritePaths=/opt/securechat

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable securechat
sudo systemctl start securechat
sudo journalctl -u securechat -f   # Ver logs en tiempo real
```

### Firewall (ufw)

```bash
sudo ufw allow 443/tcp    # HTTPS (si usas TLS nativo)
sudo ufw allow 80/tcp     # Redirección HTTP (solo con nginx)
# NO expongas el puerto 8080 públicamente si usas proxy inverso
```

### Copias de seguridad de la base de datos

```bash
# Backup online de SQLite (seguro mientras el servidor está en marcha)
sqlite3 /opt/securechat/data.db ".backup /opt/securechat/data.db.bak"

# Cron — backup diario a las 03:00
echo "0 3 * * * securechat sqlite3 /opt/securechat/data.db \".backup /opt/securechat/backups/data-\$(date +\%Y\%m\%d).db\"" \
     | sudo crontab -u securechat -
```

## 9. Lista de Verificación de Seguridad

- [ ] **Cambia `jwt.secret`** — nunca uses el valor por defecto. Genera con `openssl rand -hex 32`.
- [ ] **Activa TLS** — ya sea nativo (`tls = true`) o mediante proxy inverso con nginx.
- [ ] **Usa `mode = "private"`** salvo que quieras registro abierto de forma intencionada.
- [ ] **Cambia `federation.secret` y `federation.admin_token`** antes de activar el modo mesh.
- [ ] **Restringe el puerto 8080** en el firewall si usas proxy inverso (solo localhost debe acceder).
- [ ] **Ejecuta como usuario no-root** (Linux: el usuario de sistema `securechat` del §7).
- [ ] **Haz copias de `data.db`** regularmente — contiene todos los registros de usuarios y mensajes offline.
- [ ] **Vigila el espacio en disco** — SQLite crece con los mensajes offline; se purgan tras `offline_ttl_hours`.

## 10. Referencia de API

Todos los endpoints excepto `/api/v1/health`, `/api/v1/register`, `/api/v1/federation` y `/s2s/*` requieren un JWT válido en `Authorization: Bearer <token>`.

| Método | Ruta | Autenticación | Descripción |
|--------|------|---------------|-------------|
| `GET` | `/api/v1/health` | Ninguna | Comprobación de estado |
| `POST` | `/api/v1/register` | Ninguna | Registrar un nuevo usuario |
| `POST` | `/api/v1/invites` | JWT | Generar un código de invitación |
| `GET` | `/api/v1/users/{id}` | JWT | Obtener perfil y clave pública |
| `GET` | `/api/v1/users?q=` | JWT | Buscar usuarios (fan-out a peers en modo mesh) |
| `POST` | `/api/v1/rooms` | JWT | Crear una sala grupal |
| `GET` | `/api/v1/rooms/{id}` | JWT | Obtener información de una sala |
| `GET` | `/api/v1/rooms?q=` | JWT | Buscar salas |
| `GET` | `/api/v1/ws?token=` | JWT (query) | Upgrade a WebSocket |
| `GET` | `/api/v1/federation` | Ninguna | Info de federación y lista de peers |
| `POST` | `/api/v1/admin/federation/peers` | `X-Admin-Token` | Añadir un peer federado |
| `GET` | `/api/v1/admin/federation/peers` | `X-Admin-Token` | Listar peers federados |
| `DELETE` | `/api/v1/admin/federation/peers?url=` | `X-Admin-Token` | Eliminar un peer federado |
| `GET` | `/s2s/users/{id}` | `X-Federation-Secret` | Búsqueda S2S de usuario |
| `GET` | `/s2s/users?q=` | `X-Federation-Secret` | Búsqueda S2S de usuarios |
| `POST` | `/s2s/message` | `X-Federation-Secret` | Relay S2S de mensaje |

## 11. Resolución de Problemas

**El servidor no arranca — "open database: ..."**
La ruta de la base de datos no es escribible. Comprueba los permisos del directorio y que la ruta exista.

**Los clientes no pueden conectar — "invalid token"**
El `jwt.secret` en `config.toml` cambió después de que los usuarios se registraron. Todos los JWT existentes son inválidos. Los usuarios deben volver a registrarse o restaurar el secreto anterior.

**Las conexiones WebSocket se cortan inmediatamente**
Probablemente un timeout del proxy. Añade `proxy_read_timeout 3600s` en nginx (ver §4).

**Las llamadas de voz fallan sin audio**
El STUN falló — activa TURN (§6). Verifica también que los puertos UDP 49152-65535 están abiertos en el firewall.

**Federación — los mensajes no llegan**
1. Verifica que ambos servidores se alcanzan entre sí: `curl -H "X-Federation-Secret: <secreto>" https://peer/s2s/users/<algún_id>`
2. Comprueba que el `secret` en el registro del peer coincide con el `federation.secret` del servidor remoto.
3. Ambos servidores deben estar en modo mesh.

**"ws read error: websocket: close 1006 (abnormal closure)"**
Es una desconexión normal del cliente (app cerrada, corte de red). No es un error. El servidor lo registra solo para códigos de cierre inesperados; el 1006 ya está suprimido en la versión actual.
